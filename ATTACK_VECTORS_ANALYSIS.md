# 🎯 Registration Security: Before vs After Attack Vectors\n\n## Attack Vector 1: Frontend Role Spoofing\n\n### BEFORE (Vulnerable) ❌\n```\nAttacker opens RegisterPage\n    ↓\nSees role dropdown\n    ↓\nSelects \"ADMIN\" option\n    ↓\nFills form and clicks register\n    ↓\nFrontend sends: {name, email, password, role: \"ADMIN\"}\n    ↓\nBackend receives role=ADMIN\n    ↓\nBackend saves user with role=ADMIN\n    ↓\nAttacker IS NOW ADMIN! 🚨\n```\n\n**Defense Level: ZERO**\n- Frontend has selector (BAD)\n- No backend validation (BAD)\n- Role accepted from client (BAD)\n- Role stored as-is (BAD)\n\n---\n\n### AFTER (Secure) ✅\n```\nAttacker opens RegisterPage\n    ↓\nNo role dropdown visible\n    ↓\nOnly sees \"Student Account\" display\n    ↓\nFills form and clicks register\n    ↓\nFrontend sends: {name, email, password}\n    ↓\nBackend receives form\n    ↓\nAuthController IGNORES any role parameter\n    ↓\nUserService forces role=\"USER\"\n    ↓\nvalidateRegistrationRole() runs - no issues\n    ↓\nUser saved with role=\"USER\"\n    ↓\nAttacker only gets STUDENT role ✓\n```\n\n**Defense Level: MAXIMUM**\n- Frontend has no selector (GOOD)\n- Backend ignores client role (GOOD)\n- Backend validates role (GOOD)\n- Backend forces USER role (GOOD)\n\n---\n\n## Attack Vector 2: Direct API Bypass\n\n### BEFORE (Vulnerable) ❌\n```bash\ncurl -X POST http://localhost:8080/api/auth/register \\\n  -H \"Content-Type: application/json\" \\\n  -d '{\n    \"name\": \"Hacker\",\n    \"email\": \"hacker@test.com\",\n    \"password\": \"123456\",\n    \"role\": \"ADMIN\"\n  }'\n\n# Backend was accepting this! 🚨\n# User created with role=ADMIN\n```\n\n---\n\n### AFTER (Secure) ✅\n```bash\ncurl -X POST http://localhost:8080/api/auth/register \\\n  -H \"Content-Type: application/json\" \\\n  -d '{\n    \"name\": \"Hacker\",\n    \"email\": \"hacker@test.com\",\n    \"password\": \"123456\",\n    \"role\": \"ADMIN\"\n  }'\n\n# Backend response:\n# {\n#   \"success\": true,\n#   \"message\": \"Registration successful\",\n#   \"role\": \"USER\"  ← NOT ADMIN!\n# }\n# Role parameter was completely ignored ✓\n```\n\n**Defense:**\n- AuthController passes `null` for role (line 61)\n- UserService ignores null and forces \"USER\"\n- Validation catches invalid roles\n- Result: Attack fails silently\n\n---\n\n## Attack Vector 3: SQL Injection in Role\n\n### BEFORE (Vulnerable) ❌\n```bash\n# Attacker tries SQL injection\ncurl -X POST http://localhost:8080/api/auth/register \\\n  -d '{\n    \"role\": \"ADMIN' OR '1'='1\"\n  }'\n\n# Depending on implementation, might work\n```\n\n---\n\n### AFTER (Secure) ✅\n```bash\n# Attacker tries SQL injection\ncurl -X POST http://localhost:8080/api/auth/register \\\n  -d '{\n    \"role\": \"ADMIN' OR '1'='1\"\n  }'\n\n# Backend:\n# 1. AuthController passes null (ignores malicious role)\n# 2. Even if it didn't, validateRegistrationRole() would reject it\n# 3. Parameter binding prevents SQL injection anyway\n# Result: User created as USER with clean role ✓\n```\n\n---\n\n## Attack Vector 4: Self-Promotion via Admin Endpoint\n\n### BEFORE (Vulnerable) ❌\n```bash\n# If role update endpoint wasn't properly secured:\ncurl -X PATCH http://localhost:8080/api/users/me/role \\\n  -H \"Authorization: Bearer {user_token}\" \\\n  -d '{\"role\": \"ADMIN\"}'\n\n# If no @PreAuthorize check, user could promote themselves! 🚨\n```\n\n---\n\n### AFTER (Secure) ✅\n```bash\n# User tries to promote themselves:\ncurl -X PATCH http://localhost:8080/api/users/12345/role \\\n  -H \"Authorization: Bearer {user_token}\" \\\n  -d '{\"role\": \"ADMIN\"}'\n\n# Response: 403 Forbidden\n# Because endpoint has @PreAuthorize(\"hasRole('ADMIN')\")\n# User token has role=USER, so Spring Security rejects it ✓\n\n# Only admin token (role=ADMIN) can access this endpoint\n```\n\n---\n\n## Attack Vector 5: Token Manipulation\n\n### BEFORE (Vulnerable) ❌\n```javascript\n// Attacker opens DevTools console and tries:\nconst token = localStorage.getItem('auth_token');\n// Decode token\nconst payload = JSON.parse(atob(token.split('.')[1]));\n// Try to change role\npayload.role = 'ADMIN';\n// Re-encode\nconst fakeToken = token.split('.')[0] + '.' + btoa(JSON.stringify(payload)) + '.' + token.split('.')[2];\nlocalStorage.setItem('auth_token', fakeToken);\n\n// Problem: This changes the token signature!\n// Backend would reject it anyway, but still a risk\n```\n\n---\n\n### AFTER (Secure) ✅\n```javascript\n// Attacker tries same thing:\nconst fakeToken = 'modified_token';\nlocalStorage.setItem('auth_token', fakeToken);\n\n// Backend JWT verification:\n// 1. Signature check FAILS (token was modified)\n// 2. Even if signature was forged, backend checks JWT exp and iat\n// 3. Even if we manually craft new tokens, role comes from database\n// 4. No admin role exists in database (registration prevents creation)\n// Result: Attack fails completely ✓\n```\n\n---\n\n## Attack Vector 6: Registration Form Hijacking\n\n### BEFORE (Vulnerable) ❌\n```javascript\n// Attacker injects JavaScript on registration page:\nconst form = document.querySelector('form');\nconst oldSubmit = form.onsubmit;\n\nform.onsubmit = function(e) {\n  const roleInput = document.createElement('input');\n  roleInput.name = 'role';\n  roleInput.value = 'ADMIN';\n  form.appendChild(roleInput);\n  \n  return oldSubmit.call(this, e);\n};\n\n// Frontend now sends role=ADMIN to backend\n// Backend accepts it 🚨\n```\n\n---\n\n### AFTER (Secure) ✅\n```javascript\n// Attacker injects same JavaScript:\nconst form = document.querySelector('form');\nconst roleInput = document.createElement('input');\nroleInput.name = 'role';\nroleInput.value = 'ADMIN';\nform.appendChild(roleInput);\n\n// Even though form is hijacked:\n// 1. Frontend sends: {name, email, password, role: \"ADMIN\"}\n// 2. Backend AuthController receives request\n// 3. Backend passes null for role parameter anyway\n// 4. Even if it didn't, validateRegistrationRole() rejects ADMIN\n// 5. User created with role=USER\n// Result: Attack fails at backend layer ✓\n```\n\n---\n\n## Security Layers Summary\n\n```\n┌─────────────────────────────────────────────────────────────────┐\n│ ATTACK
